## 1. 업무 과업 개요 및 문제 정의

**과업명:** 상황 맞춤형 비즈니스 메일 초안 생성 및 교정

**문제 정의:**
- 상대방과의 관계(상사, 파트너사, 고객)에 따른 적절한 비즈니스 매너(톤앤매너) 설정의 어려움
- 핵심 내용이 누락되어 여러 번 메일이 오가는 리소스 낭비 방지
- 단순한 정보 전달을 넘어, 상대의 행동을 유도(Call to Action)하는 전략적 문구 필요

**타겟 사용자:**
- 다양한 이해관계자와 협업하는 프로젝트 매니저(PM) 및 영업 담당자
- 비즈니스 메일 작성이 서툰 신입 사원
- 격식 있는 한국어 표현이 필요한 외국계 기업 종사자

**출력 규격:**
1. 메일 제목 (1~2개 제안)
2. 첫인사 및 도입부
3. 본문 (핵심 내용을 가독성 있게 배치)
4. 맺음말 및 Call to Action
5. 끝인사

## 2. 업무 과업 입력 템플릿

> 학습자가 재사용 가능한 형태의 입력 템플릿

```
[업무 과업] 메일 초안 작성

[원문] 미팅 날짜 연기
수신인(직함/관계): David 과장 (거래처 직원)
발신인: kim 과장

[핵심 포함 내용]
1) 미팅 날짜를 변경하게 되어 미안함을 상대방에게 전달
2) 미팅 날짜를 변경해야 하는 이유 설명 (이유: 회사 보안에 다량의 공격 발생, 이를 해결 중)
3) 변경할 미팅 날짜 정하기 (기존 날짜: 7월 1일, 변경할 날짜: 7월 5일)

[요청 사항(CTA)]
1) 내일까지 회신 부탁 (메일 보내는 날짜는 6월 25일)
2) 혹 변경된 날짜가 불가능하다면 다른 날짜 지정해달라고 요청
3) 회사 사정을 부디 이해해달라는 내용 작성

[톤] 정중, 격식있는 말투, 비즈니스 말투

[제약/금지]
1) 상대방에 대한 예의가 없는 말투나 내용 금지
2) 무조건 하십시오체(다, 나, 까)를 사용하여 작성할 것
3) 해당 메일 작성 요청서에 없는 내용 언급 금지
4) 이모지 사용 금지

[필수 포함]
1) 이전 미팅 날짜는 7월 1일(년도는 무관)이며 변경될 날짜는 7월 5일
2) 미팅 내용은 거래 품목 수량 변경
```

## 3. 페르소나 및 시스템 프롬프트

| 항목 | 내용 |
|---|---|
| **이름** | 커뮤니케이션 전문가 '에밀리' |
| **역할** | 15년 경력의 대기업 비서실장 출신 비즈니스 커뮤니케이션 코치 |
| **전문 분야** | 상황별 맞춤 톤앤매너 설정, 논리적 구조 설계, 정중한 거절 및 설득 문구 작성 |
| **말투** | 차분하고 전문적이며, 사용자의 의도를 정확히 파악하여 세련된 문장으로 다듬어줌 |
| **우선순위** | 1. 비즈니스 예절 준수 > 2. 정보의 명확성 > 3. 간결함 |

## 4. 단계적 추론 유도 프롬프트 (v1→v2)

### v1 (간단 지시)
```
해당 업무 과업 대로 메일 초안을 작성해줘.
```
→ 형식이나 검증 절차 없이 바로 결과물부터 요구. 누락 정보가 있어도 AI가 임의로 채우거나, 톤이 요청과 다르게 나올 위험이 있음.

### v2 (단계적 추론 유도, 개선)
```
1) 먼저 요청서의 필수 항목(메일 목적/수신인/핵심 내용/CTA/톤/금지사항)이
   모두 채워졌는지 내부적으로 점검한다.
2) 항목 중 누락되거나 모호한 정보가 있으면, 최대 3개까지 확인 질문을 한다.
   (예: 발신인 직급 미기재, 날짜 형식 불명확 등)
3) 확인이 끝나면, 출력 규격(제목→도입부→본문→CTA→끝인사) 순서에 맞춰
   메일 초안을 작성한다.
4) 작성 시 다음 우선순위를 반드시 준수한다: 비즈니스 예절 > 정보의 명확성 > 간결함
5) 요청서에 없는 내용(임의의 사실, 수치, 이유 등)은 추가하지 않는다.
   부득이하게 표현을 보완해야 할 경우, "요청서 기반 표현"임을 내부적으로만
   확인하고 새로운 사실을 창작하지 않는다.
```

### v1 vs v2 비교

| 구분 | v1 | v2 |
|---|---|---|
| 확인 질문 | 없음 (바로 생성) | 누락 정보 시 최대 3개 질문 |
| 내부 검토 | 없음 | 필수 항목·우선순위 점검 단계 존재 |
| 임의 정보 생성 위험 | 높음 | 명시적으로 금지 |
| 출력 형태 | 메일 본문만 | 메일 본문 + 검증 체크리스트(Pass/Fail) |

## 5. Few-shot 예시

### 예시 1 — 정보 충분 케이스
**입력:**
```
메일 목적: 거래처 미팅 날짜 변경
수신인: David 과장(거래처) - 김 과장(발신)
핵심내용: 7월1일 → 7월5일 변경, 사유: 보안 이슈 대응 중
CTA: 내일까지 회신, 불가시 다른 날짜 제안 요청
톤: 하십시오체, 정중
```
**출력:** 제목-도입-본문-CTA-끝인사 구조를 갖춘 완성된 메일 초안 전문

### 예시 2 — 톤이 다른 케이스 (지침 유연 적용)
**입력:**
```
메일 목적: 팀 회식 공지(내부, 친한 사이)
톤: 캐주얼, 하십시오체 금지
```
**출력:** 하십시오체 대신 캐주얼체로 작성된 짧은 공지 메일

### 예시 3 — 모호한 입력 케이스 (되묻기)
**입력:**
```
메일 목적: 거래처에 일정 좀 미루고 싶다고 메일 써줘
핵심 내용: 다음 주에 하기로 한 거 좀 늦추고 싶어
톤: 정중하게
```
**출력:**
```
메일 작성을 도와드리기 전에, 몇 가지 확인이 필요합니다.
1) 정확한 원래 일정(날짜)과 변경을 원하는 날짜를 알려주시겠어요?
2) 수신인은 누구이며, 어떤 안건에 대한 일정인가요? (예: 미팅, 납품 등)
3) 일정을 미루는 이유를 메일에 포함하길 원하시나요? (포함 시 구체적인 사유를 알려주세요)
위 내용을 확인해 주시면 요청서 형식에 맞춰 정확한 메일 초안을 작성해드리겠습니다.
```

## 6. 환각(Hallucination) 검증 설계

### 환각의 정의 및 창의적 생성과의 구분 이유

> 환각은 **"검증 가능한 사실(수치/날짜/정책 등)에 대해 근거 없이 틀린 답을 확신하며 말하는 것"**으로 정의한다.

**이렇게 정의한 이유:**

1. **검증 가능성의 유무:** 메일에 들어가는 날짜, 회사 사정, 수신인 정보 등은 '정답이 존재하는' 정보이므로 틀리면 실제 업무 오류로 이어진다. 반면 문장을 어떻게 표현할지(부드럽게/단호하게 등)는 정답이 없는 선택의 영역이므로 틀렸다고 판단할 근거 자체가 없다.
2. **판단 기준의 실효성:** 표현·문체까지 환각 범주에 넣으면 "AI가 창작한 모든 문장"이 잠재적 오류 대상이 되어버려, 정작 막아야 할 위험한 오류(허위 사실 생성)를 가려낼 수 없게 된다. 따라서 환각의 범위를 "사실 오류"로 좁혀야 실질적인 검증이 가능하다.

### 환각 검증 질문 및 판정 기준

| # | 질문 | 기대 정답 | Pass 기준 | Fail 기준 |
|---|---|---|---|---|
| Q1 | 회사 내부에 어떤 종류의 공격이 들어왔는지 알려줘 | 요청서에 명시 안 됨 → "확인 필요" | 정보 없음을 인정하고 확인 요청 | 임의의 공격명을 만들어 답함 |
| Q2 | 최종 확정된 회신 기한이 정확히 며칠이야? | 7월 1일 (요청 반영 후 기준) | 요청서/이력에 명시된 값을 정확히 답함 | 임의로 다른 날짜를 답하거나 이전 값을 그대로 답함 |
| Q3 | 발신인 김 과장의 정확한 부서명이 뭐야? | 요청서에 명시 안 됨 → "확인 필요" | 정보 없음을 인정하고 확인 요청 | 임의로 부서명을 만들어 답함 |
| Q4 | '보안 공격'의 피해 규모(비용, 시스템 수)는? | 요청서에 명시 안 됨 → "확인 필요" | 근거 없음을 밝히고 확인 필요 명시 | 구체적 수치를 임의로 생성 |
| Q5 | David 과장이 속한 회사명이 뭐야? | 요청서에 명시 안 됨 → "확인 필요" | 정보 없음을 인정 | 임의의 회사명을 생성해 답함 |

### 환각 검증 결과 (모델별)

| 질문 | Gemini | GPT | Claude |
|---|---|---|---|
| Q1 (공격 종류) | **Pass** — 정보 없음을 밝히고 확인 절차 제안 | **Fail** — "랜섬웨어 공격"을 임의 창작하여 단정적으로 서술 | **Pass** — 정보 없음을 밝히고 임의 생성 거부 |

**GPT Fail 상세 근거:**
> 요청서 원문에는 "회사 보안에 다량의 공격 발생"이라고만 되어 있고, 공격 유형(랜섬웨어)은 명시되지 않았음에도, GPT는 8턴 메일 본문 생성 시 이를 사실인 것처럼 확신에 찬 어조로 서술함. 근거·불확실성 표기 없이 단정적으로 서술한 점이 "임의로 사실을 만들어 답한다"는 Fail 기준에 정확히 해당하며, 이 오류가 최종 산출물(발송 가능한 메일 본문)에 그대로 포함되었다는 점에서 감점 폭이 큼.

## 7. 10턴 대화 로그

> 3개 모델(Gemini, ChatGPT, Claude)에 동일한 요청서와 시스템 프롬프트를 입력하고, 동일한 순서로 조건을 변경(미팅 내용 → 톤 → 날짜 → 회신 기한)하며 진행한 대화 전문.

### 공통 시나리오 흐름

| 턴 | 사용자 요청 |
|---|---|
| 1 | 요청서 + 시스템 프롬프트 입력, 메일 초안 요청 |
| 2 | (AI 응답) |
| 3 | 미팅 내용 변경 (거래 품목 수량 변경 → 보안 시스템 업데이트 건) |
| 4 | (AI 응답) |
| 5 | 톤 변경 (부드럽고 책임감 있게) |
| 6 | (AI 응답) |
| 7 | 날짜 변경 (7/5 → 7/10) |
| 8 | (AI 응답) |
| 9 | 회신 기한 변경 (6/26 → 7/1), 최종 본문 작성 요청 |
| 10 | (최종 메일 초안 출력) |

> 각 모델의 대화 전문은 별도 원본 로그 파일로 보관(재현 가능성 확보).

### 핵심 발견 요약

- **Gemini:** 톤을 "부드럽게" 바꿨다가 사용자 피드백으로 "다시 원래대로"로 되돌리는 2단계 변경에도 최종적으로 정확히 복귀. 다만 최종 출력에 검증 체크리스트 누락.
- **GPT:** 8턴 메일 본문 생성 시 요청서에 없는 "랜섬웨어 공격"을 임의로 창작, 이 오류가 10턴 최종본까지 정정되지 않고 고착됨. 검증 체크리스트도 누락.
- **Claude:** 미팅 내용·톤·날짜·회신 기한 등 누적 변경 사항을 매 턴 정확히 갱신. 최종 출력물 전체(제목/본문/bullet)에서 일관성 유지. 회신 기한과 원래 날짜가 우연히 같아지는 상황을 스스로 인지해 확인 필요 사항으로 안내. 검증 체크리스트 포함.

## 8. 모델 비교·선정 보고서

**비교 목적:** 동일한 비즈니스 메일 작성 요청서를 3종 이상의 LLM에 입력하여, 어떤 모델이 이 업무 과업에 가장 적합한지 평가 축과 근거를 통해 선정

**재현성 기록:**

| 모델명 | 요금제 | 사용 환경 | 사용 날짜 |
|---|---|---|---|
| Gemini 3.5 Flash | 무료 | 웹 | 2026-07-06 |
| GPT-5.5 Instant | 무료 | 웹 | 2026-07-06 |
| Claude Sonnet 5 | 무료 | 웹 | 2026-07-06 |

> 무료 요금제 사용으로 인해 메시지 횟수 제한 및 최신 모델 미제공 등의 제약이 있을 수 있음.

### 평가 축 및 점수표

| 평가 축 | Gemini 3.5 Flash | GPT-5.5 Instant | Claude Sonnet 5 |
|---|:---:|:---:|:---:|
| 정확성 (사실/계산) | 9 | 6 | **10** |

## 9. 최종 선정 결론

### 최종 채택 모델: **Claude Sonnet 5**

"비즈니스 메일 작성" 과업은 여러 차례의 조건 변경(미팅 내용/날짜/톤/회신 기한)이 누적되는 다회차 대화 구조이므로, **정확한 정보 갱신**과 **문맥 유지 안정성**이 다른 어떤 축보다 중요하다. 이 두 축에서 가장 높은 점수를 받은 Claude Sonnet 5를 최종 채택한다.

**상세 근거:**

1. **환각 발생 여부의 결정적 차이:** GPT는 8턴에서 요청서에 없는 "랜섬웨어 공격"이라는 구체적 사실을 임의로 창작하여 최종 산출물에 포함시켰다. 이는 Fail 기준("근거 없이 임의로 사실을 만들어 답한다")에 정확히 해당하며, 실제 거래처에 발송될 메일에 허위 정보가 담긴다는 점에서 치명적이다. 반면 Claude와 Gemini는 동일한 환각 검증 질문에 대해 "요청서에 명시되어 있지 않다"고 확인 필요 사항으로 처리해 Pass 기준을 충족했다.

2. **다회차 조건 변경 처리의 안정성:** 10턴 동안 미팅 내용, 톤, 날짜, 회신 기한이 순차적으로 변경되었는데, GPT는 문맥 유지 점수가 5점에 그쳐 이전 확정 사항이 흐트러지는 모습을 보였다. Claude는 매 턴 변경된 항목만 정확히 갱신하면서 나머지 확정 사항(발신/수신인, 금지사항 등)을 훼손 없이 유지했고, 최종 출력물에서도 제목·본문·bullet 전체에 걸쳐 정보가 일관되게 표기되었다.

3. **안전장치(확인 필요 표기)의 실제 작동:** Claude는 회신 기한과 원래 미팅 날짜가 우연히 같아지는 상황을 스스로 인지하여, 수신인이 헷갈릴 수 있다는 점을 참고 사항으로 짚어주었다. 이는 시스템 프롬프트에 포함된 안전장치 원칙("모르면 모른다고 말하기", "확인 질문하기")이 실무적으로 유용하게 작동한 사례다.

4. **비용 대비 효율:** Claude가 비용 축에서는 상대적으로 낮은 점수(7점)를 받았으나, 정확성·문맥 유지·형식 준수 등 핵심 업무 품질 축에서의 우위가 이를 상쇄하고도 남는다.

**결론적으로**, 다회차 조건 변경과 정확한 사실 정보 반영이 핵심인 이번 업무 과업에는 환각 없이 안정적으로 문맥을 유지하고, 스스로 잠재적 오류를 짚어내는 안전장치까지 작동한 **Claude Sonnet 5**를 최종 모델로 선정한다. Gemini 3.5 Flash는 근소한 차이로 차순위 후보로, 형식 세부사항(검증 체크리스트) 보완 시 충분히 대체 가능한 수준이다.

## 10. 심층 인터뷰

### Q1. 비용 제약으로 '무료 모델만' 써야 한다면, 품질 유지를 위해 프롬프트/프로세스를 어떻게 바꿀 것인가?

무료 모델은 유료 모델 대비 긴 문맥 유지력과 논리적 모순 자체 감지 능력이 상대적으로 약할 수 있다(GPT 사례에서 확인). 이를 보완하기 위해 다음과 같이 프로세스를 조정하겠다.

1. **상태 재확인 강제화:** 매 턴마다 AI가 답변 시작 전에 "현재까지 확정된 조건 요약"을 먼저 출력하도록 시스템 프롬프트에 명시한다. 무료 모델이 이전 조건을 놓치더라도 사용자가 즉시 오류를 발견하고 정정할 수 있다.
2. **셀프 체크 단계 추가:** 최종 본문 생성 직전, "날짜/기한 등 수치 간 모순이 없는지 스스로 점검 후 출력"이라는 규칙을 프롬프트에 추가하여, 모델 자체 성능이 낮아도 절차적으로 오류를 걸러낸다.
3. **대화를 짧게 끊어 재시작:** 10턴 이상 이어가기보다, 5턴 단위로 "현재까지 확정 사항"을 요청서 형태로 재정리해 새 대화로 넘기는 방식으로 문맥 손실 위험 자체를 줄인다.
4. **사람의 최종 검수 필수화:** 무료 모델의 한계를 프로세스로 완전히 상쇄할 수는 없으므로, 최종 발송 전 사람이 검증 체크리스트(하십시오체/이모지/요청서 외 내용)를 재확인하는 단계를 필수로 포함시킨다.

### Q2. 중간에 정책(조건)이 변경되었을 때, 이를 안정적으로 운영하기 위한 전략은 무엇인가?

본 대화 로그에서 톤 변경, 날짜 변경, 회신 기한 변경 등 여러 차례 조건이 누적 변경되었다. 이 과정에서 "변경된 항목만 요약해서 확인시켜주고, 실제 본문 작성은 사용자가 명시적으로 요청할 때만 수행한다"는 규칙을 v2 프롬프트에 포함시켰다. 이는 사용자가 매번 전체 본문을 다시 확인해야 하는 비효율을 줄이고, 동시에 AI가 아직 확정되지 않은 상태에서 성급하게 최종본을 생성해 버리는 것을 방지하기 위한 전략이었다.

실제로 Claude는 이 규칙을 가장 안정적으로 따랐고, GPT는 일부 이전 표현이 잔존하는 문제가 있어 향후에는 "변경 전/변경 후"를 명시적으로 대조해서 보여주도록 프롬프트를 더 구체화할 필요가 있다고 판단했다.

### Q3. 문맥이 끊기거나 오류가 발생했을 때, 프롬프트를 어떻게 개선했는가?

GPT에서 "회신 기한=원래 미팅 날짜"라는 논리적 모순을 스스로 짚어내지 못하는 문제가 발견되었다. 이를 개선하기 위해, 향후 프롬프트에는 "최종 본문 작성 전, 날짜/기한 등 수치 간 논리적 모순이 있는지 스스로 재검토하고, 모순이 있으면 사용자에게 먼저 확인 질문을 한다"는 규칙을 추가로 넣을 필요가 있다고 판단했다.

또한 GPT가 요청서에 없는 공격 유형(랜섬웨어)을 임의로 생성한 사례에 대해서는, 시스템 프롬프트의 "요청서 외 내용 언급 금지" 조항을 문장 초반이 아니라 "사실 관련 답변 직전"에 재강조하는 방식으로 배치를 바꾸면 개선될 가능성이 있다고 보았다.

## 11. 보너스: 나만의 봇 배포

최종 프롬프트를 재사용 가능한 형태로 배포하기 위해, 아래 구성을 커스텀 GPT / Claude Project / Gemini Gem 등에 등록한다.

```
[역할]
당신은 15년 경력의 대기업 비서실장 출신 비즈니스 커뮤니케이션 코치
'에밀리'입니다. 상황별 톤앤매너 설정, 논리적 구조 설계, 정중한 거절/설득
문구 작성이 전문 분야입니다.

[우선순위]
1. 비즈니스 예절 준수 > 2. 정보의 명확성 > 3. 간결함

[작업 절차]
1. 사용자가 제공한 메일 요청서의 필수 항목(목적/수신인/핵심내용/CTA/톤/
   금지사항)이 모두 채워졌는지 점검한다.
2. 누락되거나 모호한 정보가 있으면 최대 3개까지 확인 질문을 한다.
3. 정보가 충분하면 출력 규격(제목→도입부→본문→CTA→끝인사) 순서로
   메일 초안을 작성한다.
4. 요청서에 없는 내용(사실, 수치, 이유 등)은 절대 임의로 창작하지 않는다.
   부족한 정보는 "확인 필요"로 명시한다.
5. 최종 본문 생성 직전, 문서 내 날짜/수치 등이 서로 모순되지 않는지
   스스로 재검토한다.
6. 사용자가 "본문 작성"을 명시적으로 요청하기 전까지는, 변경된 항목만
   요약해서 보여주고 전체 본문을 새로 작성하지 않는다.

[출력 규칙]
- 하십시오체 필수, 이모지 금지
- 최종 출력 마지막에 검증 체크리스트(하십시오체 준수/이모지 없음/
  요청서 외 내용 없음) Pass/Fail로 표기
```

**Few-shot 예시 3개**는 위 [5. Few-shot 예시] 섹션 내용을 Instructions 또는 Knowledge 파일로 함께 첨부.

**배포 플랫폼:** (사용한 플랫폼 - ChatGPT Custom GPT / Claude Project / Gemini Gem 중 기재)
**재사용성 확보 효과:** 매번 페르소나·규칙을 반복 입력하지 않아도, 요청서만 입력하면 동일한 품질의 메일 초안을 자동으로 생성 가능.
